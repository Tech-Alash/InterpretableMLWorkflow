Обзор проекта
========================

В репозитории реализован трёхэтапный workflow интерпретируемого машинного обучения:

1. сравнение forecasting-моделей;
2. оценка важности переменных через Shapley values и permutation importance;
3. запуск Shapley regression для статистической проверки согласованности декомпозиций с целевой переменной.

Структура репозитория
========================================

Основные каталоги и скрипты:

.. list-table::
   :header-rows: 1

   * - Путь
     - Назначение
   * - ``helpers/``
     - Общие Python-функции для конфигурации, преобразования данных, обучения моделей, агрегации прогнозов и расчёта важности признаков.
   * - ``setup_files/main.yaml``
     - Конкретная матрица экспериментов, которую читает :mod:`experiment`.
   * - ``experiment.py``
     - Главная точка входа: загружает данные, разворачивает комбинации параметров, запускает прогнозирование и считает метрики важности.
   * - ``collect_performance.py``
     - Собирает результаты рекурсивного прогнозирования в плоский CSV.
   * - ``collect_importance_forecast.py``
     - Агрегирует Shapley и permutation outputs для forecasting-сценария.
   * - ``collect_importance_out_of_bag.py``
     - Агрегирует out-of-bag результаты Shapley.
   * - ``shapley_regression.py``
     - Запускает этап Shapley regression на агрегированных forecasting-данных.
   * - ``results/``
     - Хранит сырые pickle-файлы и агрегированные CSV-артефакты.
   * - ``error_analysis.R`` and ``shapley_analysis.R``
     - Строят графики и таблицы, использованные в статье.

Как workflow устроен на практике
==================================================

Этот код проще всего воспринимать как конвейер:

* :mod:`setup_experiments` задаёт сетку параметров;
* :mod:`experiment` загружает данные и запускает матрицу экспериментов;
* :mod:`helpers.utils_predict` управляет обучением моделей, рекурсивным прогнозированием и упаковкой результатов;
* :mod:`helpers.utils_importance` считает значения Shapley и permutation-based importance;
* скрипты ``collect_*`` превращают множество pickle-файлов в удобные для анализа CSV;
* :mod:`shapley_regression` выполняет финальный этап статистического вывода.

Почему выходные файлы завязаны на hash
=========================================================

Каждая конкретная комбинация параметров получает числовой hash, сгенерированный
:func:`helpers.utils.list_experiments`. Затем этот hash встраивается в имена файлов в ``results/test_case/``.

Такой подход позволяет:

* запускать много комбинаций параметров без ручного именования файлов;
* избегать конфликтов между экспериментами;
* позже агрегировать результаты, считывая метаданные прямо из выходных файлов.

Как paper раскладывается на кодовую базу
----------------------------------------

Если смотреть на репозиторий через призму статьи, его структура становится намного понятнее.

Исследование можно мысленно разложить на несколько уровней:

* концептуальный уровень - зачем вообще нужен интерпретируемый ML workflow;
* методологический уровень - какие шаги должен пройти анализ;
* вычислительный уровень - как именно это реализуется в коде;
* аналитический уровень - как из сырых результатов получить содержательные выводы.

В кодовой базе эти уровни отражены так:

* концептуальный и методологический уровень объясняются в документации;
* вычислительный уровень реализован в ``experiment.py`` и модулях из ``helpers/``;
* аналитический уровень представлен скриптами агрегации и R-анализом.

Это важно, потому что данный репозиторий не задумывался как универсальная библиотека
с абстрактным публичным API. Он задумывался как воспроизводимая исследовательская
среда для конкретной paper-driven задачи.

Что здесь считать “главной логикой”
-----------------------------------

При первом чтении проекта полезно различать три типа файлов:

* orchestration-файлы - задают порядок шагов и управляют запуском;
* helper-модули - реализуют вычислительную механику;
* analysis outputs - переводят результаты в форму, пригодную для интерпретации.

Для этого проекта “главная логика” находится не в одном месте, а распределена:

* ``experiment.py`` отвечает за основной экспериментальный контур;
* ``helpers/utils_predict.py`` отвечает за модели, recursive forecasting и агрегацию прогнозов;
* ``helpers/utils_importance.py`` отвечает за интерпретацию моделей;
* ``collect_*.py`` и R-скрипты отвечают за упаковку результатов в исследовательские артефакты.
