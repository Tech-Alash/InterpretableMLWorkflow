Методология workflow из статьи
==============================

Эта страница пересказывает центральную методологию исследования и показывает,
как именно она отражается в коде репозитория.

Общая структура
---------------

В статье workflow состоит из трёх шагов:

1. сравнение моделей;
2. анализ важности признаков;
3. статистический вывод на основе Shapley-декомпозиций.

Важно, что это не три независимых упражнения, а последовательный pipeline.
Каждый следующий шаг имеет смысл только после предыдущего:

* если machine learning модель не даёт выигрыша по качеству, усложнять анализ нет смысла;
* если модель точнее, но её нельзя интерпретировать, её трудно использовать в прикладном процессе;
* если интерпретация есть, но нет оценки её статистической устойчивости, выводы остаются слишком хрупкими.

Шаг 1. Сравнение моделей
------------------------

Первый шаг отвечает на базовый вопрос: есть ли вообще практический смысл
использовать machine learning в данной задаче.

В логике статьи сравнение нужно для того, чтобы:

* проверить, действительно ли более сложные модели дают выигрыш относительно простого benchmark;
* понять, насколько устойчивы результаты по разным алгоритмам;
* решить, стоит ли переходить к более дорогим по времени и интерпретации этапам.

Авторы подчёркивают несколько практических моментов:

* заранее нельзя знать, какая ML-модель окажется лучшей;
* разные модели отличаются по вычислительной цене, стабильности и чувствительности к гиперпараметрам;
* использование слишком большого числа признаков может ухудшить generalization и сделать интерпретацию менее содержательной.

Как это отражено в репозитории:

* матрица сравниваемых моделей задаётся в ``setup_files/main.yaml``;
* параметры моделей и сетки поиска лежат в :mod:`helpers.utils_predict`;
* основная логика сравнения реализуется в :func:`helpers.utils_predict.run_experiments`;
* агрегированные итоги по качеству собираются скриптом :mod:`collect_performance`.

Шаг 2. Анализ важности признаков
--------------------------------

Когда найдено, что модель работает хорошо, возникает следующий вопрос:
за счёт каких переменных она делает прогноз и как именно использует информацию.

В статье проводится различие между двумя типами интерпретации:

* глобальная важность признаков - один summary по переменной на всём наборе данных;
* локальная важность - вклад признака в конкретное предсказание.

Почему в статье сделан акцент на Shapley values:

* они дают локальную декомпозицию предсказания;
* локальные вклады можно агрегировать в глобальные summary;
* у Shapley values есть сильные теоретические свойства, в частности consistency;
* это делает их удобной общей основой для объяснения machine learning моделей.

Для контраста авторы также используют permutation importance:

* permutation importance проще вычислять;
* она хорошо подходит для общего ранжирования признаков;
* но она менее богата с точки зрения локального объяснения отдельных прогнозов.

Как это отражено в репозитории:

* расчёт Shapley values и permutation importance вынесен в :mod:`helpers.utils_importance`;
* агрегирование forecast-time результатов делает :mod:`collect_importance_forecast`;
* out-of-bag вариант собирается через :mod:`collect_importance_out_of_bag`;
* итоговые CSV используются для дальнейшего анализа и графиков.

Шаг 3. Shapley regression и статистический вывод
------------------------------------------------

Обычные feature importance методы часто дают только описательное объяснение.
Они показывают, какие признаки кажутся важными, но не отвечают на вопрос,
насколько уверенно можно утверждать эту важность.

Именно поэтому третий шаг статьи особенно важен:

* Shapley-декомпозиции используются как вход в регрессионную постановку;
* на этой основе проводится статистический вывод;
* результат можно интерпретировать ближе к привычной эконометрической логике.

С практической точки зрения это нужно для того, чтобы:

* отделять устойчивые интерпретации от случайных;
* говорить о важности переменных с большей степенью уверенности;
* переводить результат machine learning анализа на язык, понятный для прикладных аналитиков.

Как это отражено в репозитории:

* центральная функция - :func:`helpers.utils_importance.shapley_regression`;
* подготовленный вход для неё создаётся после агрегации Shapley results;
* финальный запуск сделан в скрипте :mod:`shapley_regression`.

Как главы статьи соответствуют коду
-----------------------------------

Ниже практическая карта соответствия между paper и репозиторием.

.. list-table::
   :header-rows: 1

   * - Раздел статьи
     - Смысл
     - Где это в репозитории
   * - ``Introduction``
     - Мотивация: почему нужен объяснимый ML workflow
     - Документация и общий pipeline проекта
   * - ``A machine learning workflow``
     - Три шага методологии
     - ``helpers/utils_predict.py``, ``helpers/utils_importance.py``, ``shapley_regression.py``
   * - ``Data and experimental set-up``
     - Данные, target, признаки, модели, схема эксперимента
     - ``helpers/config.py``, ``setup_experiments.py``, ``setup_files/main.yaml``, ``experiment.py``
   * - ``Workflow output``
     - Качество прогноза, важность признаков, интерпретация
     - ``results/``, ``collect_*.py``, R-скрипты анализа
   * - ``Robustness``
     - Устойчивость к выбору параметров и постановки
     - Изменение YAML-конфигурации и повторный запуск экспериментов
   * - ``Discussion``
     - Практические выводы и ограничения
     - Итоговая интерпретация результатов пользователем репозитория

Что особенно важно для чтения кода
----------------------------------

Если смотреть на код без контекста статьи, может показаться, что это просто набор
скриптов для прогнозирования. Но это не совсем так. Код реализует именно исследовательскую
логику из paper:

* сначала ответить, работает ли ML лучше benchmark;
* потом понять, почему он работает;
* затем перевести это объяснение в форму, пригодную для статистического обсуждения.

То есть ценность репозитория не только в прогнозе, а в сочетании:

* ``accuracy``;
* ``interpretability``;
* ``communicability``;
* ``statistical validation``.

Именно в этом смысле репозиторий лучше читать как реализацию methodology paper,
а не только как технический pipeline.
