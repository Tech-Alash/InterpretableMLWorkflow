Результаты и интерпретация
=================================================

Workflow создаёт два основных класса выходных данных:

* сырые pickle-артефакты для каждого hash эксперимента;
* агрегированные CSV-файлы, которые проще анализировать, визуализировать и передавать в следующие скрипты.

Каталог сырых результатов
============================================

Основной каталог с сырыми результатами:

.. code-block:: text

   results/test_case/

В этом каталоге лежат:

* объекты результатов рекурсивного прогнозирования;
* out-of-bag результаты;
* forecast-time сводки по Shapley;
* forecast-time сводки по permutation importance;
* out-of-bag сводки Shapley для выбранных дат окончания train-периода.

Каталог агрегированных результатов
=================================================

Плоские агрегированные результаты лежат в:

.. code-block:: text

   results/aggregated/

Среди файлов, которые уже есть в репозитории:

.. list-table::
   :header-rows: 1

   * - Файл
     - Описание
   * - ``pred_all_test_case_raw.csv``
     - Плоская таблица forecast predictions вместе с метаданными экспериментов.
   * - ``shapley_forecast_test_case.csv``
     - Forecast-time декомпозиции Shapley, объединённые с описанием экспериментов.
   * - ``shapley_out_of_bag_test_case.csv``
     - Out-of-bag декомпозиции Shapley.
   * - ``permutation_forecast_test_case.csv``
     - Forecast-time сводки permutation importance.

Агрегация прогнозов
========================================

:mod:`collect_performance` опирается на две helper-функции:

* :func:`helpers.utils_predict.aggregate_predictions`
* :func:`helpers.utils_predict.prediction_table`

Вместе они превращают множество recursive-model outputs в единую таблицу прогнозов, привязанную к периодам.

Агрегация важности признаков
==================================================

Скрипты агрегации приводят к плоскому виду несколько разных структур:

* forecast-time Shapley results превращаются в wide-таблицы с колонками вида ``shap_<feature>``;
* forecast-time permutation results складываются в строки для absolute error, squared error и prediction deviance;
* out-of-bag результаты Shapley обрезаются до нужной bootstrap-даты перед экспортом.

Этап Shapley regression
========================================

:func:`helpers.utils_importance.shapley_regression` является финальным этапом статистического вывода.
Он сопоставляет:

* исходные наблюдаемые значения target и признаков;
* декомпозицию модельного прогноза на Shapley-компоненты.

На выходе получается таблица, которая включает:

* согласованность знаков относительно OLS-бенчмарка;
* долю Shapley;
* оценённый коэффициент регрессии;
* стандартную ошибку;
* логику односторонней проверки значимости.

Анализ на стороне R
========================================

Python workflow заканчивается на этапе, когда уже готовы чистые таблицы.
Дальше R-скрипты отвечают за presentation-oriented outputs, например:

* time-series графики observed vs predicted;
* итоговые figure по Shapley и permutation importance.

Как адаптировать workflow под другую задачу
========================================================================

Если ты хочешь применить этот workflow к другой задаче, основные точки адаптации такие:

* сам датасет и коды преобразований;
* значения в ``helpers.config``, например target, границы дат и ключевые признаки;
* матрица экспериментов в ``setup_files/main.yaml``;
* downstream analysis scripts, которые сейчас предполагают конкретные имена признаков.

Как этот раздел связан с главой ``Workflow output`` в paper
-----------------------------------------------------------

В статье раздел ``Workflow output`` отвечает не просто за демонстрацию файлов,
а за содержательную интерпретацию того, что даёт предложенный workflow на практике.

Если переложить это на репозиторий, то нужно читать outputs в таком порядке:

1. сначала понять, какая модель выигрывает по качеству;
2. затем посмотреть, какие признаки дают основной вклад;
3. потом сопоставить локальные и глобальные importance-результаты;
4. после этого переходить к Shapley regression и более строгой интерпретации.

Другими словами, отдельные CSV и pickle-файлы важны не сами по себе, а как слои
одного и того же аргумента:

* ML действительно может давать лучший прогноз;
* это преимущество связано не со случайностью, а с более гибкой структурой модели;
* эту структуру можно частично объяснить и статистически обсудить.

Как читать outputs содержательно
--------------------------------

Полезно разделять два типа чтения результатов:

* техническое чтение - проверить, что файлы создались и имеют нужную структуру;
* исследовательское чтение - понять, что они означают в терминах гипотез статьи.

Для исследовательского чтения полезны такие вопросы:

* какие модели дали лучший trade-off между качеством и интерпретируемостью;
* согласуются ли rankings признаков между разными importance measures;
* показывают ли Shapley profiles экономически правдоподобные нелинейности;
* остаются ли главные выводы стабильными при изменении постановки.

Именно этот уровень чтения и делает репозиторий исследовательским инструментом,
а не просто средством пакетной генерации CSV-файлов.
